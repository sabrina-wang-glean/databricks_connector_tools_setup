# Databricks Connector + Tools Setup Lab / Validation Toolkit
A practical, screenshot-led companion to the official documentation for configuring both the Databricks connector and Genie/SQL tools. It covers the non-obvious M2M vs. U2M split, exact setup fields, common failure modes, and some preflight validation scripts customer can run in their databricks notebooks, or their IDE.

## The most important distinction

There are **two separate Databricks integrations** in Glean with different auth paths:

| Integration | Purpose | Auth type |
|---|---|---|
| **Databricks connector** | Indexes Databricks dashboards into Glean search | M2M OAuth — service principal |
| **Databricks tools** | Genie and SQL actions in Assistant and Agents | U2M OAuth — Databricks App Connection |

A healthy, synced connector does **not** prove the Databricks tools are configured correctly.

## What access is needed

### For the Databricks connector

- A Databricks service principal and OAuth M2M credentials
- The Databricks **account-level URL** (not the workspace URL) for the Account URL field
- Account Admin is strongly preferred. A non-account-admin mode exists but has permission limitations and requires per-workspace setup.

### For Databricks tools (Genie / SQL in Assistant and Agents)

- **Account Admin access** in Databricks to create an App Connection
  - The App Connections tab lives under the **Settings gear icon** in the left sidebar at `accounts.cloud.databricks.com` — it is **not** a top-level nav item
  - If the tab does not appear at all, the user does not have account admin access and needs someone who does before proceeding
- A Databricks **workspace host URL** for the tool Host URL and OAuth endpoints
- A **Client ID and Client Secret** from the App Connection (not from a service principal — see pitfall #7)
- Tool access enabled for the right users or groups in Glean

## What goes where

### Connector fields

| Field | Value |
|---|---|
| Account URL | Full URL from the Databricks account console browser bar |
| OAuth Client ID | Service principal M2M client ID |
| OAuth Client Secret | Service principal M2M secret |

### Tool fields

| Field | Value |
|---|---|
| Host URL | `https://dbc-<instance>.cloud.databricks.com` |
| Authorization URL | `https://<workspace-host>/oidc/v1/authorize` |
| Token URL | `https://<workspace-host>/oidc/v1/token` |
| Scope | `all-apis,offline_access` |
| Client ID | Databricks App Connection client ID |
| Client Secret | Databricks App Connection client secret |

> **Callback URL:** The callback URL format is deployment-specific. Do not copy another deployment's URL. Always use the exact URL generated by your own Glean instance after saving the tool config.

## Recommended setup flow for Databricks tools

### Step 1: Create the Databricks App Connection

1. Go to `accounts.cloud.databricks.com` → **Settings → App Connections → Add Connection**
2. Set an application name (e.g. `Glean Databricks Tools OAuth`)
3. For the redirect URL — use a **placeholder** (e.g. `https://example.com`) at this stage. You need to save the Glean tool first (Step 3) to get the real callback URL. The Glean tool can be saved even before the redirect URL in Databricks is finalized.
4. Select **All APIs** (not SQL — a common mistake that silently blocks authentication)
5. **Check "Generate a Client Secret"** — this is **unchecked by default** and very easy to miss. Without it the entire tools auth flow is broken, and there is no error message pointing back to this checkbox as the cause.
6. Recommended TTLs: **360 min** for access token, **129600 min** for refresh token
   - Databricks defaults are 60 / 10080 min. The longer values reduce reauthentication churn for tools and background agent use. Align with your Databricks and Glean admins before changing.

### Step 2: Copy credentials immediately

- Copy the **Client ID** and **Client Secret** right after creation
- The secret is shown only once and cannot be retrieved later. If you close the modal without saving it, you will need to recreate the App Connection or rotate the secret.

### Step 3: Fill in the Glean tool

In **Glean Admin → Platform → Tools → Databricks Tools**:

- Enter the workspace Host URL
- Enter Client ID and Client Secret from the App Connection
- Enter Authorization URL and Token URL (workspace-level, not account-level)
- Set scope: `all-apis,offline_access`
- Optionally set a default Genie Space ID and Warehouse ID

### Step 4: Update the redirect URL in Databricks

After saving the Glean tool, copy the generated callback URL back into the Databricks App Connection's redirect URLs field, replacing the placeholder from Step 1.

> **If Databricks shows "One or more URLs are invalid":** manually type the URL instead of pasting — invisible trailing whitespace is the usual cause.

## Pre-flight validation

Validate your OAuth configuration before testing the full browser-based flow in Glean. All validation files are in the `validation/` folder.

### Option A: Databricks notebook (import into workspace)

| File | What it validates |
|---|---|
| `validation/notebooks/databricks_oauth_app_validation.py` | Connector (M2M): OIDC discovery + service principal token |
| `validation/notebooks/databricks_tools_oauth_validation.py` | Tools (U2M): OIDC discovery + App Connection token + SQL smoke test + Genie API test |

All values entered via **widget fields** at the top — no hardcoding. Run all cells top to bottom.

### Option B: Standalone Python script (no Databricks required)

```bash
# Install dependency
pip install requests

# Connector (M2M)
python validation/scripts/validate_connector_oauth.py \
  --workspace-host https://dbc-xxxx.cloud.databricks.com \
  --client-id YOUR_CLIENT_ID \
  --client-secret YOUR_CLIENT_SECRET

# Tools (U2M App Connection)
python validation/scripts/validate_tools_oauth.py \
  --workspace-host https://dbc-xxxx.cloud.databricks.com \
  --client-id YOUR_CLIENT_ID \
  --client-secret YOUR_CLIENT_SECRET
```

> **Important:** A PASS on the M2M token test confirms the App Connection exists and the secret is correct. It does **not** prove the full U2M (browser redirect) flow will work. Also verify the redirect URL in Databricks exactly matches the callback URL Glean generated.

## Functional test in Glean

Pre-flight validation only confirms the OAuth configuration — it does **not** replace an end-to-end test in Glean. After completing the setup, validate the integration with a simple functional test.

The easiest approach is a scratch Agent:

1. Create a new Agent.
2. Add a single text input named `query`.
3. Add a Databricks step using either **Search Databricks with Genie** or **Search Databricks with SQL**.
4. Pass `[[query]]` into the Databricks step.
5. Add a **Respond** step and test the Agent in Preview.

Alternatively, test the Databricks tool directly in **Glean Assistant**.

## Common pitfalls

### 1. Mixing up connector auth and tool auth
Connector uses M2M (service principal). Tools use U2M (App Connection). A healthy connector sync does **not** validate the tools OAuth flow.

### 2. Using the wrong URL type
Connector uses the account URL. Tools use the **workspace host URL** and workspace-level OAuth endpoints.

### 3. Selecting SQL instead of All APIs
When creating the App Connection, select **All APIs**. SQL scope blocks tool authentication.

### 4. Missing the "Generate a Client Secret" checkbox
This checkbox is **unchecked by default**. Without it the tools auth flow is completely broken, with no error message pointing back to this as the cause. Always verify it is checked before saving.

### 5. Losing the client secret
The secret is displayed once only. If lost, you need to create a new App Connection or rotate the secret via the Databricks CLI.

### 6. Redirect URL validation error
If Databricks shows "One or more URLs are invalid" even for a structurally valid URL:
- **Manually type** the URL instead of pasting — invisible trailing whitespace is the usual cause
- Or create the App Connection with `https://example.com` first, then update after Glean generates the real callback URL

### 7. "OAuth application not available in Databricks account"
This error means the Client ID is not recognized as an App Connection for U2M:
- A service principal OAuth secret can pass an M2M token test but still fail here — **a passing M2M test does NOT confirm an App Connection exists**
- These are two different credential types in Databricks
- Fix: confirm the exact `client_id` is listed under Account Console → Settings → App Connections (not just under Service Principals)

### 8. App Connections tab not visible
Only **account admins** can see the App Connections tab. If it's missing, the user needs to find someone with account admin access.

### 9. Backend action flags not enabled
Enabling the Databricks tools requires **Glean-internal deployment config changes** that cannot be made by customers or SAs:

```ini
# Required for Genie action
databricks_genie.enabled = true
additionalSupportedActions = ...,querygenie

# Required for direct SQL action
additionalSupportedActions = ...,databricksexecutesql
```

If these flags are not set, the actions will not appear even after OAuth is configured correctly. Contact the Glean team that owns deployment configs.
